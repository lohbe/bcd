## Worked example
<!-- The worked example / code goes here or in a sibling file -->
no worked example, but some discussion around it was produced. since OKF is a spec based implementation of karpathy's llm-wiki, i was curious how to maintain a consistent OKF across a set of 'bundles' once it becomes large enough beyond a human. so i asked claude:

Would OKF: https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md, be a bit difficult to handle without LLM automation pipeline, once it becomes big enough? Like for example in the front matter yaml, `type:` is not really a fixed type, and some sort of entity resolution would be required

and it confirms:

Yes — I'd say that's a real pain point, and it's basically baked into the design on purpose. A few things stand out from the actual spec:

**`type` is explicitly free-text by design.** Type values are not registered centrally, producers should pick values that are descriptive and self-explanatory, and consumers must tolerate unknown types gracefully. So you'll get `BigQuery Table`, `bq_table`, `Table (BigQuery)`, `BigQuery table`, etc. all meaning the same thing once you have multiple teams or bots contributing. There's no enum, no registry, nothing to validate against. That's a deliberate non-goal too — defining a fixed taxonomy of concept types is explicitly listed as a non-goal.

**Conformance checking won't save you.** The formal conformance bar is very low: a bundle is conformant if every non-reserved markdown file has parseable YAML frontmatter, that frontmatter has a non-empty type field, and reserved files follow the index/log structure. A linter can catch "missing type," but it can't catch "this type is a near-duplicate of that type." That's a semantic problem, not a syntactic one — you're right that it needs entity resolution / clustering, not schema validation.

**Extension keys make it worse.** Producers may include any additional keys, and consumers should preserve unknown keys and not reject documents with unrecognized fields. So beyond `type`, any producer-defined key is fair game too — you could get drift in `tags` vocabulary, inconsistent `resource` URI schemes, etc. Same fuzzy-matching problem, just at a different field.

**Links compound the issue.** The specific kind of relationship a link expresses is conveyed by prose, not by the link itself, and consumers that build a graph view typically treat all links as untyped directed edges. So even your relationship graph is untyped by default — if you want "joins-with" vs "depends-on" vs "parent-of" as distinct edges, you're back to inferring that from surrounding text, which again points at an LLM pass rather than a parser.

So at small scale (one team, one bundle, careful humans) you can probably keep `type` vocabulary tidy by convention. Once you're at the scale where multiple teams or agents are generating concepts independently — which is explicitly the target use case (the format is designed to be authored by people, generated by agents, exchanged across organizations, and consumed by both) — you'll almost certainly need a canonicalization/entity-resolution step: embedding-based clustering of `type` strings, an LLM pass to propose a canonical taxonomy, maybe a periodic "merge these 40 near-duplicate types into 12" cleanup job. The spec is explicit that this is on the consumer's plate, not something it will ever standardize — that permissiveness is presented as a feature (this permissive consumption model is intentional — OKF is meant to remain useful as bundles grow, get refactored, and are partially generated by agents), but it really is punting the hard part downstream to whoever builds the consumption/indexing layer.

## Reflection
<!-- What did you learn? What was hard? -->
I learnt that the tradeoffs architects encounter pre-LLM, also appear even in this LLM era. In the sense that the more flexible your design is, the more you push down the complexity to be managed to subsequent users. To manage this, the OKF system cannot remain passive, and requires more processing steps to resolve duplicate types, checking for semantic inconsistencies, etc. Essentially, different approach (e.g. graph rag vs agents/okf) prioritises different aspects, and one will deal with different side effects.
